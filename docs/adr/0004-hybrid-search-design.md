# ADR-0004: ハイブリッド検索の設計（pg_bigm + pgvector + RRF）

## ステータス

採用

## コンテキスト

記憶の検索方式として以下の選択肢があった：
- キーワード検索のみ
- ベクトル検索のみ
- ハイブリッド検索（両方の統合）

## 決定

**pg_bigm（キーワード） + pgvector（ベクトル） + RRF（統合）** のハイブリッド検索を採用。

## 理由

- **キーワード検索のみ** では「意味的に近いが表現が異なる」記憶を見つけられない
- **ベクトル検索のみ** では完全一致のキーワードが含まれていても低スコアになることがある
- **ハイブリッド** は両方の強みを活かせる。RRFで公平に統合することで、片方だけにヒットした記憶も、両方にヒットした記憶も適切にランキングできる

### 各コンポーネントの選定

| コンポーネント | 選定 | 理由 |
|--------------|------|------|
| キーワード検索 | pg_bigm | 日本語bigramで形態素解析不要。PostgreSQL拡張として統合 |
| ベクトル検索 | pgvector (HNSW) | PostgreSQL内でベクトル検索が完結。別サービス不要 |
| 統合方式 | RRF (k=60) | 異なるスコア体系のランキングを公平に統合。実装がシンプル |
| 時間減衰 | 指数減衰 (半減期30日) | 新しい記憶を優先しつつ、古い記憶も検索可能に保つ |
| 埋め込み | multilingual-e5-small (384次元) | ローカル実行、多言語対応、コンパクト |
| 重複排除 | コサイン類似度 >= 0.90 | 保存前に最近傍1件と比較。ほぼ同一内容の二重保存を防止 |

## パラメータの根拠

| パラメータ | 値 | 根拠 |
|-----------|-----|------|
| RRF k | 60 | 論文「Reciprocal Rank Fusion outperforms Condorcet and individual Rank Learning Methods」の推奨値 |
| 時間減衰半減期 | 30日 | 月単位のプロジェクトサイクルに合わせた。設計判断の鮮度は1ヶ月で半減する想定 |
| 重複閾値 | 0.90 | 競合（engram等）の採用値に合わせ、意味的に同等な記憶の重複保存を抑制する水準 |
| maxResults | 20 | コンテキストウィンドウの圧迫を避けつつ、十分な候補を返す |

## 影響

- PostgreSQLに2つの拡張（pgvector, pg_bigm）が必要 → カスタムDockerイメージで対応
- 検索時に2つのクエリが並列実行される → レイテンシは遅い方に依存
- RRFと時間減衰の計算はアプリケーション層で実行
