# ADR-0006: 記憶の容量管理戦略

## ステータス

採用

## コンテキスト

claude-memory の記憶が無制限に蓄積されると、検索性能の低下やストレージコスト増大を招く。容量管理の仕組みが必要だが、以下の方針を決定する必要があった：

1. **削除戦略**: どの記憶を優先的に削除するか
2. **上限値**: 何件を上限とするか
3. **自動実行タイミング**: いつ容量管理を実行するか

### 検討した選択肢

#### 削除戦略

- **LRU（Least Recently Used）**: 最終アクセス日時が古い順に削除 — 一度だけ大量にアクセスされた記憶が残り続ける
- **LFU（Least Frequently Used）**: アクセス回数が少ない順に削除 — 実際の利用頻度を反映
- **TTL（Time To Live）**: 作成から一定期間で削除 — 古くても重要な記憶を失う

#### 上限値

- **1,000件**: 少なすぎて頻繁にpruneが走る
- **10,000件**: pgvectorのHNSWインデックスで十分な性能を維持できる規模
- **100,000件**: インデックスサイズが大きくなり、メモリ使用量が増大

## 決定

### LFU戦略を採用

`leastAccessed` 戦略として、`access_count` 昇順 + `last_accessed_at` 昇順で削除対象を選定する。

### 上限10,000件

- pgvectorのHNSWインデックスは10,000件規模で検索レイテンシ数ms以内を維持
- 個人の記憶管理ツールとして十分な容量
- `countAll()` で現在の件数を取得し、超過分を削除

### 保存時の自動prune

`memory_save` 実行時に `countAll()` で件数を確認し、10,000件を超過している場合は超過分を `deleteLeastAccessed` で自動削除する。手動の `memory_cleanup` ツールでも任意に実行可能。

## 理由

### なぜLFUか

LRUは「最後にアクセスした時刻」だけを見るため、偶然1回アクセスされた記憶と日常的に参照される記憶を区別できない。LFUなら `access_count` によって真に利用頻度の低い記憶を特定できる。同一アクセス回数の場合は `last_accessed_at` で二次ソートし、より長期間参照されていない記憶を優先削除する。

### なぜ10,000件か

- pgvectorのHNSWインデックスは数万件までリニアにスケールし、10,000件ならインデックス構築・検索ともに実用的な速度を維持
- 1日に数十件の記憶を保存する想定で、数ヶ月〜1年分の蓄積に相当
- 上限を超えた分だけ削除するため、急激なデータロスが発生しない

### なぜ保存時の自動pruneか

- ユーザーが明示的にクリーンアップを実行する必要がない
- 保存のたびに件数チェックするオーバーヘッドは `COUNT(*)` 1回で軽微
- 上限超過時のみ削除が走るため、通常の保存操作への影響は最小限

## 影響

- `StorageRepository` に `countAll()` と `deleteLeastAccessed(limit)` メソッドを追加
- `SaveMemoryUseCase` に保存後の自動prune処理を追加
- `memory_cleanup` ツールに `leastAccessed` 戦略を追加（手動実行用）
- 既存の `lastAccessedOlderThan` 戦略は維持（日数ベースの削除も引き続き利用可能）
