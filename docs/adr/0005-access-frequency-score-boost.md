# ADR-0005: アクセス頻度によるスコアブースト

## ステータス

採用

## コンテキスト

検索スコアリングは RRF + 時間減衰の2要素で構成されていたが、「頻繁にアクセスされる記憶はユーザーにとって重要度が高い」というシグナルを反映できていなかった。アクセス頻度を検索スコアに組み込む方法として以下の選択肢を検討した：

1. **加算方式**: `score + f(accessCount)` — スコア体系のスケールに依存し、調整が難しい
2. **乗算方式**: `score * f(accessCount)` — 元のスコアの相対的な差を保持したまま頻度で重み付け
3. **ランキング要素として RRF に追加**: 3つ目のランキングとして統合 — 実装が複雑

## 決定

**乗算方式** を採用し、以下の計算式でスコアブーストを適用する：

```
boostedScore = decayedScore * (1 + min(accessCount / 50, 0.2))
```

- 除数: 50
- 最大ブースト倍率: 1.2倍（accessCount >= 50 でキャップ）
- 適用タイミング: 時間減衰の後

## 理由

### なぜ乗算方式か

加算方式は RRF スコア（0.01〜0.03程度の小さな値）に対して固定値を足すため、元のスコアの相対差が崩れやすい。乗算方式なら元のスコア差を維持したまま、頻繁にアクセスされる記憶を一律に押し上げられる。

### なぜ除数 50 か

- 1日1回程度のアクセスで約2ヶ月弱で最大ブーストに到達する
- これは「本当に継続的に使われている記憶」を識別するのに妥当な期間
- 数回のアクセスでは微小なブーストに留まり、ノイズにならない

### なぜ最大 1.2 倍にキャップするか

- ブーストが大きすぎると、内容の関連性よりアクセス頻度が支配的になる
- 1.2倍（20%増）は「同程度の関連性なら頻繁にアクセスされる方を優先」という程度の影響力
- 明らかに関連性が低い記憶が頻度だけで上位に来ることを防止

## 影響

- `Memory` エンティティに `accessCount` フィールドを追加
- `StorageRepository` に `access_count` カラムを追加（マイグレーション 0002）
- 検索ヒット時に `accessCount` をインクリメント（`lastAccessedAt` の更新と同時）
- upsert 時に既存の `accessCount` を保持（新規値で上書きしない）
