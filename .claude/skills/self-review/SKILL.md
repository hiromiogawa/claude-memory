# セルフレビュー

実装が完了したら、以下の検証サイクルを実行する。全パスするまでコミットしない。

## 検証サイクル

1. **lint**: `pnpm lint`
   - OXLint + Biome のエラーを修正
   - Biome の auto-fix が TypeScript の型エラーを引き起こす場合があるため、修正後に型チェックも確認

2. **test**: `pnpm test` または対象パッケージのみ `pnpm --filter <pkg> test`
   - 失敗したテストは修正してから次へ

3. **dep-check**: `pnpm dep-check`
   - 依存方向違反があれば import を修正

4. **knip**: `pnpm knip`
   - 未使用の export, ファイル, 依存があれば削除

5. **e2e-coverage**: E2E テストカバレッジの確認（判断項目）
   - 変更が以下の層にかかっているか確認:
     - **インフラ層**: storage / repository / DB スキーマ / ドライバ接続 / 外部API接続
     - **ユースケース層**: use-cases / ビジネスロジック
     - **外部接続**: embedding プロバイダ等、モックでは検出できない結合部
   - 該当する場合、既存の E2E テストがその変更パスをカバーしているか確認
   - カバーされていなければ、対応する E2E テストケースを追加してからコミットする
   - **スキップしてよいケース**:
     - プレゼンテーション層のみの変更（ハンドラの出力フォーマット等）
     - ドキュメントのみの変更
     - テストコードのみの変更
     - lint / 型定義のみの変更

### なぜ必要か

ユニットテストはモックで通過しても、実 DB やドライバとの結合で初めて発覚するバグがある（例: `Date` オブジェクトをドライバに直接渡すバグ、SQL テンプレートの型不一致、スキーマ不整合など）。このチェックで E2E カバレッジを自然に伸ばし、同種のバグを防ぐ。

## 修正ループ

```
実装 → lint → 失敗? → 修正 → lint（再実行）
                ↓ 成功
            test → 失敗? → 修正 → test（再実行）
                ↓ 成功
            dep-check → 失敗? → 修正 → dep-check（再実行）
                ↓ 成功
            knip → 失敗? → 修正 → knip（再実行）
                ↓ 成功
            e2e-coverage → 不足? → E2Eテスト追加 → test（再実行）
                ↓ 十分
            コミット可能
```

## 注意事項

- パッケージ単体で作業している場合も、最終確認はルートの `pnpm test` で全体テストを実行する
- pre-commit フックで lint-staged + knip + dep-check が自動実行されるが、コミット前に手動で確認しておくとフック失敗による手戻りを防げる
- e2e-coverage は自動ツールではなく判断項目。変更の性質を見て必要性を判断する
