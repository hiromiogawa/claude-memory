# 長期記憶の活用（claude-memory MCP）

## セッション開始時（必須）

セッション開始時に以下の検索を **必ず** 実行し、過去の文脈を把握する：

1. `memory_search` でプロジェクト名や現在の作業内容を検索
2. `memory_search` でユーザーの好みを検索（tags: preference）
3. 検索結果を踏まえて作業を開始する

## 保存ルール

以下の情報はユーザーに確認せず `memory_save` で自動保存する：

| 保存する情報 | タグ |
|-------------|------|
| 設計判断とその理由（「なぜAではなくBを選んだか」） | `design-decision` |
| ユーザーの好み・作業スタイル | `preference` |
| バグの原因と解決策 | `bug` |
| プロジェクト固有の知識（アーキテクチャ制約、運用ルール） | `rule`, `architecture` |
| 議論の結論や合意事項 | `design-decision` |

**保存しないもの:**
- 一般的な技術知識（公式ドキュメントに書いてあること）
- 一時的な作業状態
- コードそのもの

## タグ体系

保存時は必ず tags を付ける（検索精度に直結）：

- `design-decision` — 設計判断
- `preference` — ユーザーの好み
- `bug` — バグ原因・解決策
- `rule` — プロジェクトルール
- `architecture` — アーキテクチャ知識
- `process` — 開発プロセスの知見

## 作業中の検索

- **設計判断時** — 過去に同様の判断をしていないか検索
- **バグ遭遇時** — 同じバグの解決策がないか検索
- **新技術導入時** — 過去の技術選定理由を検索

## ADR連携

設計判断があった場合は以下の **両方** に記録する：
1. `docs/adr/` にADRファイルを作成（人間が読む構造化ドキュメント）
2. `memory_save` で長期記憶に保存（AIが検索するインデックス）

## メンテナンス

- `memory_stats` で記憶の総数を定期確認
- `memory_cleanup` で90日以上未アクセスの記憶をクリーンアップ（dry-run で確認後に実行）
- `memory_export` で定期バックアップ
